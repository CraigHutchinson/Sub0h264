# L1.1 — Row-based decode+deblock streaming pipeline

**Layer:** L1 Architecture | **Expected gain:** +2–3 fps ball, +15–20% deblock-bound content | **Risk:** Low | **Complexity:** M | **Status:** pending

## Context

Deblock is 13.8% of ESP32 frame time (~11.5 ms @ 12 fps). On ESP32-P4 the
frame buffer lives in PSRAM (200 MHz). The current two-pass decode-then-
deblock ordering re-reads the entire 460 KB luma+chroma frame from PSRAM
during the deblock pass, since the decode pass evicted those cache lines.

## Problem

[decoder.hpp](../../../components/sub0h264/src/decoder.hpp) — `decodeSlice`
MB loop (around lines 642–865) runs to completion, then a separate deblock
pass (around lines 890–906) iterates every MB edge reading `yRow()`/`uRow()`/
`vRow()`. Every read touches PSRAM because the L1 working set is ~32 KB and
a full 640×368 frame is ~460 KB.

## Options

### Option A — Row-granular deblock after each MB row
After MB row `r` finishes decoding, immediately deblock row `r−1` (its top
edges now depend on row `r−1` + row `r−2`, both recently written so hot in
L1/L2). Two MB rows (~20 KB luma + chroma) stay cached.

- **Pro:** minimal state change, no API disruption; clean regression checkpoint
- **Pro:** all cross-row dependencies satisfied (row `r−1` is complete + row
  `r` is decoded before we deblock its top edge)
- **Con:** last row needs a trailing flush after loop exits
- **Con:** slice boundary handling (mid-frame slices) requires careful
  row-boundary tracking

### Option B — MB-granular interleaved deblock
After each MB finishes decoding, deblock its left + top edges immediately.

- **Pro:** finest cache locality (only the just-decoded MB + its left/top
  neighbours are in cache)
- **Con:** right + bottom edges can only be deblocked when the NEXT MB
  arrives — more bookkeeping
- **Con:** harder to reason about `disable_deblocking_filter_idc == 1`
  (filter disabled at slice boundary) because we'd need to track filter
  state across interleaved MB+edge traversal

### Option C — Tile-based (multi-row blocks)
Process 2+ MB rows at a time in larger super-rows.

- **Pro:** better for wide frames (e.g. 4K)
- **Con:** more code for a marginal win at 640×368; premature for current
  target resolution

## Chosen approach: **Option A**

Option A is the smallest diff for the largest cache-pressure reduction at
our target resolution. Option B is strictly better asymptotically but the
slice-boundary + right-edge bookkeeping complexity isn't justified yet.
Revisit B if Option A is insufficient.

## Implementation

**Files:**
- [components/sub0h264/src/decoder.hpp](../../../components/sub0h264/src/decoder.hpp) — `decodeSlice` MB loop and deblock pass
- [components/sub0h264/src/deblock.hpp](../../../components/sub0h264/src/deblock.hpp) — no changes expected; existing `deblockMbLuma`/`deblockMbChroma` already take per-MB args

**Changes:**
1. Track the current MB row (`currentMbRow = firstMb / widthInMbs_`).
2. After decoding every MB, when we transition to a new row `r > 0`, call
   `deblockMbRow(r − 1)` helper that iterates `mbX = 0..widthInMbs_-1` and
   deblocks each MB's horizontal + vertical edges.
3. After the decode loop finishes the last row, call `deblockMbRow(last)`.
4. Extract the existing deblock pass into `deblockMbRow(row)` helper.

**Pseudocode:**
```cpp
uint32_t prevRow = UINT32_MAX;
for (uint32_t mbIdx = firstMb; mbIdx < lastMb; ++mbIdx) {
    uint32_t r = mbIdx / widthInMbs_;
    if (prevRow != UINT32_MAX && r != prevRow)
        deblockMbRow(prevRow);
    decodeMb(mbIdx);
    prevRow = r;
}
if (prevRow != UINT32_MAX) deblockMbRow(prevRow);
```

## Validation

**Must remain 99 dB bit-exact** (`ctest --preset default` + `python scripts/
yuv_diff.py psnr ours.yuv jm.yuv <w> <h>`):
- Tapo C110 (119 frames, 640×368 High CABAC)
- All 6 wstress fixtures
- wstress_p8x8_sub (10 frames)
- All synthetic CABAC IDR / I+P fixtures in `test_synthetic_quality.cpp`

**Unit tests:** existing deblock unit tests must pass unchanged.
**Edge case fixtures:** single-row frame (`cabac_16mb_wide_row.h264`),
multi-slice frame (if available — else add one), frame-boundary MVs.
**Bench target:** Ball-Base-640 ESP32 fps ≥ prev + 2.0.

## Risks & mitigations

- **Last-row flush:** easy to forget. Unit test: decode a single-row frame
  and confirm PSNR = 99 dB.
- **Multi-slice frame boundaries:** if `decodeSlice` is re-entered mid-frame,
  `prevRow` state resets and we could double-deblock a row. Mitigate by
  tracking "deblocked up to row R" at the Decoder level, not per-slice.
- **`disable_deblocking_filter_idc == 1`:** slice-boundary filter gating
  must still work — the row-granular scheme doesn't affect this since the
  filter is per-edge, not per-row.
