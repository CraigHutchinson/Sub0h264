# L1.2 — Eliminate `currentFrame_` duplication

**Layer:** L1 Architecture | **Expected gain:** +1–2% | **Risk:** Low | **Complexity:** S | **Status:** pending

## Context

The decoder holds both `currentFrame_` (a standalone Frame member) and
`activeFrame_` (pointer to either `currentFrame_` or a DPB slot). When they
point at different buffers, a 460 KB memcpy runs per frame to copy decoded
pixels out of `currentFrame_` into the DPB target. This is vestigial
infrastructure from before the DPB was wired as the direct decode target.

## Problem

[decoder.hpp](../../../components/sub0h264/src/decoder.hpp) lines 546–551
and 912–926. The `TODO` note at line 532 already calls out the duplication.
Per-frame impact: 460 KB × PSRAM-bound memcpy ≈ 2–3 ms on ESP32-P4.

## Options

### Option A — Always decode into the DPB target
Assign `activeFrame_ = dpb_.allocate(...)` at slice start; remove
`currentFrame_` member entirely. Intra-frame / pre-decode bookkeeping that
currently uses `currentFrame_` as scratch moves to the DPB target directly.

- **Pro:** eliminates the memcpy unconditionally
- **Pro:** simplifies mental model (single active buffer)
- **Con:** any failure mid-decode leaves a partial frame in the DPB — but
  we already mark slots invalid on error, so this is fine
- **Con:** touches several "emit frame" paths; must verify all writers flush
  to DPB not to `currentFrame_`

### Option B — Keep `currentFrame_`, lazy-elide memcpy when pointers match
Only memcpy when `activeFrame_ != &currentFrame_`. This is what the code
effectively does already — partial fix only.

- **Pro:** trivial
- **Con:** vestigial member remains; the memcpy still fires in the "different
  buffer" path that this opportunity is supposed to eliminate

## Chosen approach: **Option A**

The only reason to keep `currentFrame_` is if some error path depends on
having an untouched backup, and we already roll back via DPB slot state.
Option A is the right answer and matches the existing TODO.

## Implementation

**Files:** [components/sub0h264/src/decoder.hpp](../../../components/sub0h264/src/decoder.hpp)

**Changes:**
1. Remove the `currentFrame_` member.
2. At slice-start, set `activeFrame_` = target DPB Frame pointer directly.
3. Delete the fallback memcpy block at lines ~912–926.
4. Audit every `currentFrame_` reference: each becomes `*activeFrame_`.
5. The `currentFrame()` public accessor keeps its name but returns
   `activeFrame_` (or the most recently completed DPB frame).

**Sizeof check:** `sizeof(Decoder)` should drop by `sizeof(Frame)` ≈ 460 KB
when Decoder holds allocations. Verify with `static_assert` after change.

## Validation

- `ctest --preset default` all tests pass
- 99 dB bit-exact on Tapo + wstress + wstress_p8x8_sub
- Desktop shootout: no fps regression (we expect improvement)
- ESP32 build: verify heap footprint reduction via `heap_caps_get_info`

## Risks & mitigations

- **Intra-prediction neighbour reads:** intra prediction reads samples from
  the frame buffer. Those reads previously went through `*activeFrame_`
  (which was `currentFrame_`). Verify they still resolve correctly.
- **Frame callback timing:** whoever consumes the decoded Frame sees a DPB
  slot directly. The DPB slot reference must remain stable until the next
  slice begins. This is already the contract, but check callback order.

## Follow-ups / compounding opportunities

- Pairs with [L3.2 frame alignment](L3.2_frame_alignment.md) — with only
  DPB-backed frames, the alignment decision is made in one place (DPB
  allocator).
- Reduces decoder constructor cost (large Frame member gone).
