# L1.3 — Static max-bounded context arrays

**Layer:** L1 Architecture | **Expected gain:** +1% | **Risk:** Low | **Complexity:** S | **Status:** pending

## Context

Per-frame decode resizes 6 context arrays (`nnzLuma_`, `nnzCb_`, `nnzCr_`,
`mbMotion_`, `mbIntra4x4Modes_`, `cabacNeighbor_`) via `std::vector::resize`
or equivalent. Total ≈ 284 KB. On ESP32-P4 this hits PSRAM and can trigger
allocator work even when sizes don't change between frames (constant
resolution).

## Problem

[decoder.hpp](../../../components/sub0h264/src/decoder.hpp) lines 557–562.
Resize is only technically needed when `widthInMbs_` / `heightInMbs_`
change (new SPS with different dimensions). For the common case of a
fixed-resolution stream, we pay the resize cost every frame anyway.

## Options

### Option A — `std::array` sized at `cMaxWidth * cMaxHeight / 256` MBs
Allocate for the max supported resolution (640×480 = 1200 MBs) at
construction. Index operations are identical. Heap churn eliminated.

- **Pro:** zero runtime allocation; cache-friendly (contiguous, fixed addr)
- **Pro:** `static_assert` at compile time that array is big enough
- **Con:** always pays 284 KB even if stream is 320×240 — but we're on
  ESP32-P4 with 32 MB PSRAM; trivial.
- **Con:** dictates a hard resolution cap (`cMaxWidth`/`cMaxHeight` in
  [sps.hpp](../../../components/sub0h264/src/sps.hpp))

### Option B — Keep `std::vector`, skip resize when size matches
Check `if (new_size != cur_size) resize(new_size)`.

- **Pro:** trivial
- **Con:** doesn't eliminate the initial resize allocation; still requires
  a heap-backed vector
- **Con:** doesn't help cache locality (vector storage may be re-allocated
  by the allocator on first use)

### Option C — Reserve once at first SPS, then only resize if SPS changes
Track last-seen (w,h); skip resize if unchanged.

- **Pro:** reasonable middle ground
- **Con:** still vector-backed; Option A is strictly simpler

## Chosen approach: **Option A**

We already have `cMaxWidth = 640` and `cMaxHeight = 480` as hard caps. 284 KB
of static allocation is nothing on the target platform. Flat arrays give us
the best cache behaviour.

## Implementation

**Files:** [components/sub0h264/src/decoder.hpp](../../../components/sub0h264/src/decoder.hpp)

**Changes:**
1. Compute `constexpr uint32_t cMaxMbs = (cMaxWidth / 16U) * (cMaxHeight / 16U);`
   in [sps.hpp](../../../components/sub0h264/src/sps.hpp).
2. Replace each `std::vector<...>` member with `std::array<..., N>` where N
   is the max-MB derivative (e.g. `mbMotion_` → `std::array<MbMotionInfo,
   cMaxMbs * 16>`).
3. Delete the `resize` calls; replace with `std::fill` over the active
   range `[0, totalMbs)` / `[0, totalMbs * 16)` at slice-start.
4. Update `sizeof(Decoder)` expectation; add `static_assert(sizeof(Decoder)
   <= cMaxDecoderSize)` to catch accidental growth.

## Validation

- `ctest --preset default` passes
- 99 dB bit-exact on golden fixtures
- ESP32 heap footprint report shows the expected shift (heap → static/BSS)
- Constructor timing: `std::array` construction of large arrays can be
  slow; measure `Decoder::Decoder()` time before/after — should be
  equivalent (zero-init is cheap) or a wash

## Risks & mitigations

- **Stack overflow if Decoder is on stack:** unlikely given current sizes,
  but add `static_assert` guard. Enforce heap-only construction via
  `std::make_unique<Decoder>` everywhere (already the case in test apps).
- **Larger BSS on ESP32:** 284 KB in PSRAM is fine; verify the linker
  places these arrays where we want them (`.bss` in PSRAM, not `.iram`).

## Follow-ups / compounding opportunities

- Simpler integration with [L3.4 mb context packing](L3.4_mb_context_packing.md)
  — flat-array of `MbContext` struct is the natural extension.
- Deterministic memory layout helps with ESP32-P4 cache predictability.
