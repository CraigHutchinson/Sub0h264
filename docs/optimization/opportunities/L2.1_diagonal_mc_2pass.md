# L2.1 — Diagonal MC 2-pass separable filter ⭐ HIGHEST IMPACT

**Layer:** L2 Algorithm | **Expected gain:** +3–4 fps ball (25–30% of MC cost) | **Risk:** Medium | **Complexity:** L | **Status:** pending

## Context

Motion compensation is 67.7% of decode time on ESP32-P4 (56.4 ms/frame at
12 fps). The 9 diagonal quarter-pel positions (§8.4.2.2.1) currently invoke
`j2D` per output pixel, which in turn does 6×6 = 36 `getSample` calls each
with 2 clamp branches. For a 16×16 block at position (2,2): 256 × 36 = 9 216
samples, every one with bounds-checked reads.

The ball benchmark randomly exercises all 16 fractional positions, so the
expensive diagonals dominate. Other benchmarks (scroll, still, Tapo) use
only a handful of positions.

## Problem

[inter_pred.hpp](../../../components/sub0h264/src/inter_pred.hpp) lines
226–322 dispatch to `j2D` inline per pixel for diagonal positions. The 6-tap
filter is applied twice (horizontal then vertical) for every single output
pixel instead of computing an intermediate row pass that can be reused by
multiple output columns.

Spec §8.4.2.2.1: diagonal positions require an **unclipped intermediate**
— the horizontal 6-tap result feeds into the vertical 6-tap, and the final
`(sum + 512) >> 10` clip only happens once at the end. This is the key
invariant any separable-filter implementation must preserve.

## Options

### Option A — Two-pass separable 6-tap with `int16_t` intermediate buffer
1. Pass 1: for each row of the output region plus 5 rows of top/bottom
   margin (height + 5), compute horizontal 6-tap into an `int16_t` temp
   buffer of dimension `(width) × (height + 5)`. No clip.
2. Pass 2: for each output column, apply vertical 6-tap on the intermediate
   buffer with the final `(sum + 512) >> 10` clip.

For a 16×16 block: 21 rows × 16 samples horizontal = 336 multiply-accumulate
per row; 16 columns × 16 samples vertical = same on the intermediate. Total
~6 000 MACs vs 9 216 before — plus massively better cache behaviour.

- **Pro:** exactly the structure the spec encourages; straightforward to
  validate bit-exactness
- **Pro:** sets up L4.5 (getSample elimination) because all bounds checks
  are done once during the horizontal pass margin
- **Pro:** the intermediate buffer is small (`(16+5) × 21 × 2 B = 882 B`)
  — fits easily on stack
- **Con:** L-sized refactor in `inter_pred.hpp`; many positions to cover
- **Con:** must dispatch per fractional position to share the right
  intermediate (see sub-cases below)

### Option B — Just cache the intermediate for the 4 "expensive" quarter-pel
Implement only for positions e, f, i, p (and the other diagonals by
rotation). Leave half-pel diagonals untouched.

- **Pro:** smaller diff
- **Con:** misses the biggest gain (j position) and requires duplication

## Chosen approach: **Option A with per-position dispatch**

Specialise the 9 diagonal positions into their sub-cases:
- `b` (half-pel horizontal) — 1-pass horizontal 6-tap
- `h` (half-pel vertical)   — 1-pass vertical 6-tap
- `j` (half-pel diag)       — 2-pass, both passes 6-tap
- `a, c` (quarter-horizontal) — avg of integer + `b`
- `d, n` (quarter-vertical)   — avg of integer + `h`
- `e, g, p, r` (quarter-diag corners) — avg of `b` and `h` at integer
- `f, i, k, q` (quarter-diag edges)   — avg of `b` and `j`, or `h` and `j`

For quarter-pel positions that need `j`, reuse cached 2-pass intermediate.

## Implementation

**Files:** [components/sub0h264/src/inter_pred.hpp](../../../components/sub0h264/src/inter_pred.hpp)

**Changes:**
1. Define `constexpr int16_t cMcFilter[6] = { 1, -5, 20, 20, -5, 1 };`
2. Add helpers:
   - `horz6Tap(src, dst, w, h)` — write int16_t intermediates
   - `vert6TapClip(src_i16, dst_u8, w, h)` — write final clipped u8
   - `vert6Tap(src_i16, dst_i16, w, h)` — no-clip intermediate for `j`
3. Dispatcher switch on (dx_frac, dy_frac):
   - Fetch reference region with 5-pixel border via a single clamp-aware
     copy (sets up L3.1)
   - Compute b/h/j intermediates only for the cases the position needs
   - Final averaging + clip per spec §8.4.2.2.1
4. Keep the current per-pixel dispatch as a `#if 0`'d fallback for a
   build-gated fallback during the transition.

**Bit-exactness invariant:** intermediate buffer is `int16_t`, no clipping
until the final step. Reference implementation stays available in a
separate function for one-for-one diffing during development.

## Validation

**Critical** — this change modifies the core MC pipeline:
- Bit-exact unit test in `tests/test_inter_pred.cpp` for all 16 fractional
  positions against a reference implementation (slow per-pixel), over a
  grid of input blocks. Any mismatch → revert or fix.
- 99 dB PSNR on Tapo, all wstress, wstress_p8x8_sub, ball_* benchmarks.
- Bench target: Ball-Base-640 fps + ≥ 3.0 on ESP32.

## Risks & mitigations

- **Integer overflow in intermediate:** 6-tap on uint8 input with
  `1,-5,20,20,-5,1` max positive sum is `20*2*255 = 10200`, max negative
  is `-5*2*255 = -2550`. Fits in int16_t (`[-32768, 32767]`) — OK.
- **Off-by-one in margin:** 6-tap needs 5 pixels of context on each side.
  Horizontal pass needs +5 height (for vertical tap context), vertical pass
  needs +5 width on intermediate. Explicit table of required margin per
  position.
- **Memory footprint:** stack buffer for 16×16 is 16 × 21 × 2 B = 672 B.
  `alignas(16)` for ESP32-P4 vector load alignment.
