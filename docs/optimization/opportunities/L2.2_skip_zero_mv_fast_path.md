# L2.2 — Skip-MB / zero-MV fast paths

**Layer:** L2 Algorithm | **Expected gain:** +0.5–1 fps (still/scroll), marginal on ball | **Risk:** Low | **Complexity:** S | **Status:** pending

## Context

Skip MBs and zero-MV coded MBs are extremely common in low-motion content
(still, scroll, most of Tapo). Their MC is a pure copy from the reference:
no fractional-pel filter, no interpolation, no intermediate. But the
current path still goes through the generic `lumaMotionComp` /
`chromaMotionComp` dispatcher, which branches on fractional position
before taking the integer-copy branch.

## Problem

Two inefficiencies:

1. [inter_pred.hpp](../../../components/sub0h264/src/inter_pred.hpp) —
   `chromaMotionComp` doesn't short-circuit for `dx==0 && dy==0`. Chroma
   bilinear still computes `(8-dx)(8-dy)/64` weighted averages with all
   terms zeroed out for dx=0,dy=0 — wasted multiplies.
2. [decoder.hpp](../../../components/sub0h264/src/decoder.hpp) —
   `decodePSkipMb` calls the full MC pipeline even when the inferred MV
   is `(0,0)` (very common when both neighbours are unavailable or have
   zero MVs), which would be a pure memcpy.

## Options

### Option A — Explicit early-out at dispatch
Add `if (mvX == 0 && mvY == 0) { blockCopy(...); return; }` at the top of
both `lumaMotionComp` and `chromaMotionComp`.

- **Pro:** trivial and localised
- **Pro:** the compiler may already do branch elimination when inlined,
  but explicit is safer on in-order ESP32-P4
- **Con:** adds a hot-path check for every MC call, including when it
  doesn't help. Negligible (one compare + branch predicted-correct ~99%).

### Option B — Separate `pSkipMb` fast path at decoder level
When the MB is skipped AND both A/B neighbours give zero MV, skip
`lumaMotionComp`/`chromaMotionComp` entirely — do a raw `blockCopy` from
reference into target.

- **Pro:** also skips the MV-prediction arithmetic
- **Con:** need to double-check the derived MV before entering fast path
  (skipMb with derived zero-MV happens often but not always)
- **Con:** more code paths to maintain

### Option C — Both A and B (A for the general case, B for the skip case)
Layered: A gives you zero-MV coded MBs; B gives you zero-MV skip MBs on
top of A.

## Chosen approach: **Option C — both**

Two-liner in the MC functions (Option A) + explicit zero-MV memcpy path in
`decodePSkipMb` (Option B). Cheap, low-risk, easy to measure.

## Implementation

**Files:**
- [components/sub0h264/src/inter_pred.hpp](../../../components/sub0h264/src/inter_pred.hpp)
- [components/sub0h264/src/decoder.hpp](../../../components/sub0h264/src/decoder.hpp) (`decodePSkipMb`)

**Changes:**
1. `lumaMotionComp`: early-return with `blockCopy(refBlock, dstBlock, 16, 16)`
   when `mvX == 0 && mvY == 0`.
2. `chromaMotionComp`: same, with `8,8` block size.
3. `decodePSkipMb`: after MV prediction, if `derivedMv.x == 0 && .y == 0`,
   call a fast path that copies luma (16×16) + chroma (8×8 × 2) from ref.
4. Skip BS/deblock computation for zero-MV skip MBs where possible (Bs=0
   anyway when both partitions have matching refIdx + MV).

## Validation

- 99 dB bit-exact on all golden fixtures
- Unit test: decode a fixture with guaranteed zero-MV coded and skip MBs
  (e.g. `static_scene_baseline.h264` already used in test_synthetic_quality)
- Bench: Still-Base-640 fps + ≥ 0.5; Tapo fps improvement (Tapo has many
  skip MBs in low-motion regions)

## Risks & mitigations

- **Sub-pixel precision of zero MV:** `mvX == 0 && mvY == 0` in quarter-pel
  units covers integer-aligned zero. If the MV is e.g. (0, 0) but
  weighted prediction applies, the fast path must not bypass weighting.
  Check `weightedPredFlag` first; only take fast path when disabled.
- **Chroma MV scaling:** chroma MV is luma MV / 2 (for 4:2:0). If luma MV
  is 0, chroma MV is 0 — safe.
- **Reference frame boundary:** integer MV can still touch reference
  boundary if MB position is at frame edge. Use the existing clamped
  copy helper, not raw memcpy.

## Follow-ups / compounding opportunities

- Benefits grow when combined with [L3.3 direct-to-frame MC](L3.3_direct_frame_mc.md)
  — zero-MV + zero-CBP is a pure raw memcpy
- Pairs with [L2.3 BS cache](L2.3_deblock_bs_cache.md) — zero-MV skip MBs
  often have Bs=0 cached edges
