# L2.3 — Deblocking BS cache per-MB

**Layer:** L2 Algorithm | **Expected gain:** +0.5–1 fps (20–30% of deblock time) | **Risk:** Low | **Complexity:** M | **Status:** pending

## Context

The deblocking filter recomputes **boundary strength (BS)** per edge during
the deblock pass. BS depends on whether the edge crosses an MB boundary,
whether neighbours are intra vs inter, CBP bits, motion vector differences,
and reference frame indices — all information the MB decoder already has
when decoding each MB.

Per spec §8.7.2.1 Table 8-17, BS can be one of {0, 1, 2, 3, 4}. Each 16×16
MB has 8 vertical + 8 horizontal edges of 4-pixel segments on its interior
and boundary, total 16 × 4 = 64 BS values per MB. Chroma edges are derived
from luma (§8.7.2.2) so we can store luma only.

## Problem

[decoder.hpp](../../../components/sub0h264/src/decoder.hpp) /
[deblock.hpp](../../../components/sub0h264/src/deblock.hpp) — BS is
recomputed in the deblock pass at lines 342–362, 406–424, 473–491, 536–556.
This requires re-fetching per-MB context (intra flag, CBP, MVs, refIdx)
for every edge, which is cache-hostile.

## Options

### Option A — Cache full BS per MB per edge (uint8_t[totalMbs * 16])
Store 16 BS values per MB (4 bits each fits in a nibble, but align to
uint8_t for simpler indexing). 1200 MBs × 16 = 19 200 bytes.

- **Pro:** deblock pass reads a precomputed byte instead of chasing
  MV/CBP/refIdx arrays
- **Pro:** fits in L1 cache (~20 KB vs 32 KB L1)
- **Con:** 16 edges × `uint8_t` is 16 B per MB; could be packed to 8 B
  (nibbles) for better density — Option A1 below

### Option A1 — Packed 4-bit nibbles
`uint8_t bs[totalMbs * 8]` — each byte holds 2 edges. Total 9 600 bytes.

- **Pro:** fits 2× more MB contexts in cache
- **Con:** shift/mask overhead on read

### Option B — Cache only Bs==0 / non-zero bit per edge
1 bit per edge = 2 bytes per MB. Deblock pass re-derives the actual BS
value when bit is set (still saves a branch for all-zero cases).

- **Pro:** smallest memory
- **Con:** partial win only

## Chosen approach: **Option A (full BS, one byte per edge)**

The simplest win with biggest uplift. A1 (nibbles) is a potential follow-up
if cache pressure becomes a concern at larger resolutions.

## Implementation

**Files:**
- [components/sub0h264/src/decoder.hpp](../../../components/sub0h264/src/decoder.hpp) — add `bsCache_` member array, write during MB decode
- [components/sub0h264/src/deblock.hpp](../../../components/sub0h264/src/deblock.hpp) — add `bs` parameter to deblock helpers; they read from the cache instead of computing

**Changes:**
1. Add member: `std::array<uint8_t, cMaxMbs * 16> bsCache_;` (16 = 4 top
   edges + 4 left edges + 4 internal vert + 4 internal horiz).
2. During `decodeMb`, at the end, compute BS for all 16 edges of this MB
   using the fresh per-MB context (still in registers / L1 cache):
   - top edge (4 × 4-pixel segments)
   - left edge (4 × 4-pixel segments)
   - internal vertical edges (4 × 4)
   - internal horizontal edges (4 × 4)
3. Store into `bsCache_[mbIdx * 16 .. mbIdx * 16 + 16)`.
4. Update deblock helpers (`deblockMbLuma`, `deblockMbChroma`) to accept a
   pointer to this MB's 16-entry BS slice and use it directly — remove
   the in-loop BS derivation code.
5. Chroma BS: derived from luma Bs per §8.7.2.2 (`BsChroma = BsLuma` for
   chroma 4:2:0) — reuse the cached luma array, no extra storage.

## Validation

- `ctest --preset default` passes
- 99 dB bit-exact on golden fixtures
- Unit test: explicit BS-computation test that decodes an MB, reads the
  cached BS array, and compares against a reference computation on the
  same MB context — ensures write-during-decode path matches the old
  read-during-deblock logic
- Bench: Scroll, Tapo fps + ≥ 0.3 (deblock-bound)

## Risks & mitigations

- **BS depends on NEIGHBOUR data** that may not yet be decoded when the
  current MB is finishing — specifically, the edge ON the right and bottom
  boundaries of the current MB requires the NEXT MB (raster-order) to be
  decoded before BS can be computed.
  - **Mitigation:** only compute BS for edges that depend on already-decoded
    neighbours:
    - Top edge: depends on current MB + top neighbour (both decoded) ✓
    - Left edge: depends on current MB + left neighbour (both decoded) ✓
    - Internal edges: depend only on current MB ✓
    - **The right/bottom edges belong to the NEXT MB in raster order**
  - This matches the natural deblock pass ordering, so no issue.

- **Pairs naturally with [L1.1 row-based deblock](L1.1_row_based_deblock.md):**
  once a row is fully decoded, all its internal + top + left BS values
  are ready; the bottom edge waits for the next row to start (at which
  point its top edge = this row's bottom edge, and it IS ready).

## Follow-ups / compounding opportunities

- Pairs with [L1.1 row-based deblock](L1.1_row_based_deblock.md) — BS
  cache is computed during decode of a row, consumed during deblock of
  the same row, all in L1
- Unlocks [L3.4 mb context packing](L3.4_mb_context_packing.md) — BS
  slice can be part of the MbContext struct
- Simpler A1 packing follow-up if cache pressure becomes an issue
