# L2.4 — Horizontal-edge column-major transpose

**Layer:** L2 Algorithm | **Expected gain:** +0.5 fps (10–15% of deblock) | **Risk:** Low | **Complexity:** M | **Status:** pending

## Context

Deblocking horizontal edges filters 4-pixel column segments that span 8
rows (4 above the edge, 4 below). Each filter tap reads one pixel from
each of 8 consecutive rows — 8 different `yRow(r)` pointers, each a
separate cache-line access in the worst case.

Vertical edges don't have this problem: they filter 4 consecutive samples
along a single row (all in one cache line).

## Problem

[deblock.hpp](../../../components/sub0h264/src/deblock.hpp) lines 495–524
— horizontal edge filter iterates `r = 0..7` and calls `yRow(py + r)` 8
times per 4-pixel column. On ESP32-P4 with 32-byte cache lines and
frame stride > 32 B, each row is a different cache line → 8 cache-line
touches per 4-pixel column, 32 per horizontal edge.

## Options

### Option A — Transpose a 16×8 neighbourhood into a local scratch buffer
Copy the 16-wide × 8-tall region (MB width plus top/bottom margin) into a
column-major local buffer. Filter it as if it were a vertical edge
(sequential memory access). Transpose back to the frame.

- **Pro:** converts 8 scattered reads to 8 sequential reads (within one
  scratch buffer row, hot in L1)
- **Pro:** filter kernel is the same as vertical edge — reuse code
- **Con:** 128 bytes of transpose + 128 bytes of de-transpose per edge

### Option B — Compute filter outputs 4 rows at a time with SIMD-style SWAR
Pack 4 rows of 4 pixels (16 bytes) into a uint64_t pair; filter using
SWAR tricks; unpack.

- **Pro:** theoretically fastest
- **Con:** hard to get right; PIE is the proper target for this on ESP32

### Option C — No change; accept the cache thrashing
- **Pro:** zero effort
- **Con:** 10–15% of deblock time on the table

## Chosen approach: **Option A**

Transpose-based filtering is a known technique. The 16×8 scratch buffer is
tiny (128 B, fits in registers on a proper SIMD ISA; two cache lines on
RISC-V). Option B should wait for PIE availability.

## Implementation

**Files:** [components/sub0h264/src/deblock.hpp](../../../components/sub0h264/src/deblock.hpp)

**Changes:**
1. Add helper: `transpose8x16_u8(const uint8_t* src, uint32_t srcStride,
   uint8_t* dst)` and its inverse.
2. For horizontal edge filter:
   - Gather: read 8 rows × 16 columns from the frame (cache-line aligned
     if possible) via transpose into a 16×8 scratch buffer (column-major).
   - Filter: apply the existing vertical-edge filter to the scratch
     buffer (now sequential access).
   - Scatter: transpose scratch back to frame.
3. Keep the gather scratch `alignas(16)` for ESP32-P4 vector load.

**Pseudocode:**
```cpp
alignas(16) uint8_t scratch[16 * 8];  // 16 cols, 8 rows column-major
transpose16x8(frame.yRow(py - 4) + px, frame.stride(), scratch);
// Now filter scratch as if it were vertical edges
for (int col = 0; col < 16; col += 4)
    filterVerticalEdge(&scratch[col * 8 + 4], 1, bs[col/4]);
transpose8x16(scratch, frame.yRow(py - 4) + px, frame.stride());
```

## Validation

- 99 dB bit-exact on golden fixtures
- Explicit transpose round-trip unit test
- Bench: any deblock-bound fixture should show + ≥ 0.3 fps

## Risks & mitigations

- **Stride mismatch:** transpose must use the frame's actual stride
  (luma stride ≠ width when cropped). Pass stride explicitly.
- **Chroma path:** chroma horizontal edges are 8×4 not 16×4; needs its
  own transpose helper or a parameterised version.
- **Edge disabled by BS=0:** don't transpose if entire edge row is Bs=0
  (already the current fast path).
