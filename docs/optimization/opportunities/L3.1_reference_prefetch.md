# L3.1 — Reference prefetch into stack buffer ⭐ SECOND HIGHEST IMPACT

**Layer:** L3 Data flow | **Expected gain:** +2 fps on ball; eliminates MC PSRAM pressure | **Risk:** Low | **Complexity:** M | **Status:** pending

## Context

MC is 67.7% of decode time. The reference frame is in PSRAM (460 KB at
640×480 4:2:0). Fractional-pel filter taps read a `(16+5) × (16+5) = 21×21`
luma region per 16×16 output block — up to 441 scattered PSRAM reads per
MB, each clamped for frame-edge cases. Cache lines don't help much because
the reads are stride-dependent (hop 640 bytes per row).

## Problem

[inter_pred.hpp](../../../components/sub0h264/src/inter_pred.hpp)
`lumaMotionComp` / `chromaMotionComp` — filter inner loops call
`getSample(ref, x + dx, y + dy)` which does bounds-clamped pointer
dereference into PSRAM. The bounds check runs ~441 times per MB; the
PSRAM reads are the dominant cost on ESP32-P4.

## Options

### Option A — Copy reference region into stack scratch, then filter from scratch
Before MC dispatch, copy the `(16+5) × (16+5) = 441 B` reference region
into a `int16_t` or `uint8_t` stack buffer. Apply frame-edge clamping
once during the copy (replicating the last in-frame pixel for out-of-
frame positions). The filter inner loops then read from the scratch
buffer — guaranteed in L1.

Sizes:
- Luma 16×16 MV: 21 × 21 × 1 B = 441 B scratch
- Luma 16×8 / 8×16 / 8×8 partitions: smaller scratch (13×21 or 13×13)
- Chroma 8×8: 9×9 = 81 B scratch
- Rounded up to cache-line alignment: `alignas(32)` doubles the footprint
  but keeps it well under 2 KB stack

### Option B — Row-wise prefetch (refill intermediate every row)
Prefetch 21 pixels per row, process the row, move on. Less stack pressure,
more function calls.

- **Pro:** smaller peak stack
- **Con:** we gain less — the scattered reads become the row fetch, but
  the clamp checks still run per-row (vs once up-front in Option A)

### Option C — Full 32×32 super-block prefetch (for 32×32 aggregation)
Prefetch two MBs' worth of reference. More locality at the cost of stack.

- **Pro:** future-proof
- **Con:** 32×37 > 1 KB scratch; not worth it for 16×16 today

## Chosen approach: **Option A**

Stack-allocated rectangular prefetch with up-front clamping is the classic
approach (matches JM, ffmpeg's asm paths, x264's reference). 441 B is
trivial on ESP32-P4 stack.

## Implementation

**Files:** [components/sub0h264/src/inter_pred.hpp](../../../components/sub0h264/src/inter_pred.hpp)

**Changes:**
1. Add helper `prefetchRefLuma16x16(frame, srcX, srcY, scratch)` that
   copies the 21×21 region. For integer-MV positions, copy 16×16 only.
2. Clamp coordinates at copy-time: `int sx = clamp(srcX + col, 0, w-1);
   scratch[row * stride + col] = ref.yRow(sy)[sx];`
3. Change the filter functions to take a `const uint8_t* scratch` with
   known stride and interior dimensions — no more `getSample` calls.
4. Same for chroma with 9×9 scratch (chroma is bilinear, 2-tap not 6-tap).

**Stack footprint:** per-MB MC path adds ~512 B of scratch. ESP32-P4 has
ample stack (>8 KB) on the decode task.

**Alignment:** `alignas(32)` for ESP32-P4 cache-line + vector load.

## Validation

- 99 dB bit-exact on golden fixtures (any MV near frame edge must still
  produce correct clamped output)
- Unit test: frame-edge MV cases
  - MB at (0,0) with MV (-3, -3)
  - MB at right edge with MV (+3, 0)
  - MB at bottom-right with MV (+3, +3)
- Bench: Ball-Base-640 + ≥ 2.0 fps on ESP32, Tapo any improvement

## Risks & mitigations

- **Edge clamping correctness:** the spec (§8.4.2.2.1) specifies sample
  access with clamping; the scratch buffer must embed exactly the same
  clamped samples. Test with frame-edge MVs.
- **Interaction with weighted prediction:** weighted-pred applies AFTER
  interpolation; scratch-based interpolation output feeds into it the
  same way. No change required.
- **Stack depth:** if multiple scratch buffers stack up (e.g. per
  partition), measure max stack depth on ESP32.

## Follow-ups / compounding opportunities

- **Direct enabler of [L4.5 getSample elimination](L4.5_getsample_elimination.md)**
  — once scratch is in play, the `getSample` bounds-check inside the
  filter loop can be deleted entirely. +1.5 fps compounding.
- **Prerequisite for [L2.1 diagonal MC 2-pass](L2.1_diagonal_mc_2pass.md)**
  — the 2-pass filter operates on a region; scratch is the natural home.
- Pairs with [L3.3 direct-to-frame MC](L3.3_direct_frame_mc.md).
