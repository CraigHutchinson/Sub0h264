# L3.2 — Frame buffer cache-line alignment

**Layer:** L3 Data flow | **Expected gain:** +5–8% overall (compound) | **Risk:** Low | **Complexity:** S | **Status:** pending

## Context

ESP32-P4 has 32-byte cache lines. Standard `std::vector<uint8_t>` uses
default allocator alignment (typically 8 or 16 bytes). When the frame
buffer's base address + stride aren't cache-line aligned, MB column
boundaries straddle cache lines — doubling the cache-line touches for
edge MBs on every decode pass.

## Problem

[frame.hpp](../../../components/sub0h264/src/frame.hpp) `allocate` (lines
~37–51) uses `std::vector::resize`, which delegates alignment to the
default allocator. Frame stride is `widthInMbs_ * 16` (640 for 640×480,
which happens to be cache-aligned, but not guaranteed at other widths).

## Options

### Option A — `std::unique_ptr<uint8_t[]>` + `std::aligned_alloc(32, size)`
Explicit aligned allocation with a custom deleter that routes back to
`std::free` (aligned_alloc's complement).

- **Pro:** standard library, portable
- **Pro:** stride aligned to 32 bytes automatically if width is
  multiple of 32 — pad width to 32 B if needed
- **Con:** no more `vector<uint8_t>` conveniences (`.data()`, `.size()`)
  but those aren't really needed for a frame buffer

### Option B — Custom aligned allocator on `std::vector<uint8_t, AlignedAllocator<32>>`
Keep `std::vector`, swap the allocator.

- **Pro:** minimal call-site changes
- **Con:** `AlignedAllocator` has to implement the full Allocator
  concept (~50 lines); easy to get wrong

### Option C — On ESP32, use `heap_caps_aligned_alloc(32, size, MALLOC_CAP_SPIRAM)`
Platform-specific but gives explicit PSRAM placement.

- **Pro:** most precise control of where the buffer lives
- **Con:** desktop build needs `std::aligned_alloc` fallback; two paths

## Chosen approach: **Option A with Option C fallback on ESP32**

Use a platform-neutral `AlignedBuffer` wrapper class in `frame.hpp`:
- Desktop / non-IDF: `std::aligned_alloc(32, n)` + custom deleter
- ESP-IDF: `heap_caps_aligned_alloc(32, n, MALLOC_CAP_SPIRAM)` when the
  size indicates a frame buffer; `MALLOC_CAP_INTERNAL` otherwise

## Implementation

**Files:**
- [components/sub0h264/src/frame.hpp](../../../components/sub0h264/src/frame.hpp) — allocation path
- Possibly new file `components/sub0h264/src/aligned_buffer.hpp`

**Changes:**
1. Add `AlignedBuffer` RAII wrapper around `aligned_alloc` with 32-byte
   alignment; custom deleter calls `free` / `heap_caps_free`.
2. In `Frame::allocate`, replace `std::vector<uint8_t>` members with
   `AlignedBuffer`. Preserve the `.yRow(r)` / `.uRow(r)` / `.vRow(r)`
   API — callers shouldn't care.
3. Pad stride to 32 B if `widthInMbs_ * 16 % 32 != 0` (this affects
   widths like 80, 144, 336 that aren't multiples of 32 — 640 and 320
   are already fine, so Tapo and existing fixtures unaffected).

## Validation

- 99 dB bit-exact on golden fixtures
- `static_assert(alignof(decltype(*yRow(0))) >= 32)` or similar runtime
  assert
- ESP32 `heap_caps_get_info` shows expected PSRAM usage
- Bench: compound benefit with other optimisations — measure at the end
  of the optimisation phase sequence, not standalone (effect is small
  individually but compounds)

## Risks & mitigations

- **Deleter mismatch:** `aligned_alloc` complement is `free`. On Windows
  MSVC, use `_aligned_malloc`/`_aligned_free` pair. Platform shim needed.
- **Stride change:** if we pad stride, every call-site that assumes
  `stride == width` must go through `frame.stride()`. Audit.
- **Test fixtures:** all fixtures at widths multiple of 32 today, so this
  doesn't change observable output in tests — but a future fixture at
  e.g. 336 × 240 (CIF derivative) would exercise the padding logic.
  Add such a fixture to the test matrix.

## Follow-ups / compounding opportunities

- Prerequisite for PIE SIMD paths ([L5.3 catalog](L5.3_pie_simd_catalog.md))
  — 128-bit aligned loads/stores require 16-byte alignment min
- Pairs with [L3.1 reference prefetch](L3.1_reference_prefetch.md) —
  aligned source makes the scratch copy faster
