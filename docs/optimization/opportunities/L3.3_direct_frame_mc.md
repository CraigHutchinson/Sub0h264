# L3.3 — Direct-to-frame MC for zero-CBP MBs

**Layer:** L3 Data flow | **Expected gain:** +0.5 fps | **Risk:** Low | **Complexity:** S | **Status:** pending

## Context

Skip MBs and zero-residual coded MBs (CBP == 0) have no residual to add —
MC output IS the final pixel value. Currently the decoder MCs into a
stack `uint8_t predLuma[256]` then `memcpy`s into the output frame after
the (empty) residual add. 384 B of wasted memcpy per MB.

## Problem

[decoder.hpp](../../../components/sub0h264/src/decoder.hpp) CABAC inter
path around line 3196 (and CAVLC equivalent):
```cpp
uint8_t predLuma[256];
lumaMotionComp(ref, predLuma, ...);
// ... residual decode ...
if (cbpLuma == 0) {
    memcpy(frame.yMb(mx, my), predLuma, 256);
}
// else add residual and write
```

For CBP == 0 (very common), the intermediate buffer is strictly wasted.

## Options

### Option A — Check CBP BEFORE MC; if zero, MC directly into target
Swap the order: peek at CBP (which requires decoding it), and if it's
zero, pass the target frame address directly to MC instead of the stack
scratch.

- **Pro:** saves the memcpy unconditionally for CBP==0 MBs
- **Con:** CBP decode is interleaved with residual in the current flow
  (`coded_block_pattern` is decoded early, before residual). Good — we
  can make this decision early.

### Option B — MC into scratch always; skip memcpy when CBP==0 AND residual
has nothing to add
Equivalent to saying "only memcpy when you'd otherwise skip the residual
add loop". The current code already does this implicitly for zero-coeff
blocks, but still does `memcpy(scratch → frame)` for the prediction.

- **Pro:** minimal refactor
- **Con:** misses the point — the stack buffer + memcpy pair is the cost
  we want to eliminate

### Option C — Union: MC output pointer is either `predLuma` or
`frame.yMb()` based on CBP
Single function takes an output pointer; caller chooses based on CBP.

- **Pro:** cleanest API
- **Con:** identical outcome to Option A; naming question only

## Chosen approach: **Option A / C**

MC takes an output pointer. Caller passes `frame.yMb(mx, my)` when
`cbpLuma == 0`, else a stack scratch. Requires decoding CBP before MC,
which matches the natural slice-data order anyway.

## Implementation

**Files:**
- [components/sub0h264/src/decoder.hpp](../../../components/sub0h264/src/decoder.hpp) CABAC + CAVLC inter paths
- [components/sub0h264/src/inter_pred.hpp](../../../components/sub0h264/src/inter_pred.hpp) — `lumaMotionComp` already takes an output pointer; we just pass the frame address instead of scratch

**Changes:**
1. Reorder inter MB decode: after MV prediction and CBP decode, branch on
   `cbpLuma == 0`:
   - Zero: call `lumaMotionComp` with `out = frame.yMb(mx, my)`, skip
     residual stage entirely.
   - Non-zero: unchanged path (scratch → residual add → frame).
2. Same for chroma with `cbpChroma == 0`.
3. Care with stride: frame `yMb` takes `yStride`; stack scratch uses 16
   stride. Both are already explicit parameters to MC.

## Validation

- 99 dB bit-exact on golden fixtures (skip/zero-CBP MBs are everywhere in
  scroll/still/Tapo content)
- Unit test: decode a fixture with `static_scene_baseline.h264` (heavy
  skip) and confirm bit-identical output
- Bench: Still, Scroll +0.3 fps; Tapo small bump

## Risks & mitigations

- **Partial CBP** (e.g. cbpLuma==0 but cbpChroma≠0 or vice versa): handle
  luma and chroma independently. Easy.
- **Weighted prediction:** weighting applies during MC; writes the
  weighted value to the output pointer. Compatible with direct-to-frame.
- **Deblock dependency:** the next MB's deblock reads these pixels.
  Direct-to-frame is fine because the pixels are written before deblock
  runs (even with [L1.1 row-based deblock](L1.1_row_based_deblock.md)).
