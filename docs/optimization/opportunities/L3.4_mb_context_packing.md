# L3.4 — Per-MB context struct packing

**Layer:** L3 Data flow | **Expected gain:** +2–3% | **Risk:** Low | **Complexity:** M | **Status:** pending

## Context

Deblocking, CABAC neighbour lookup, and intra-prediction neighbour
derivation all touch per-MB context. Today those fields live in
separate arrays (`nnzLuma_`, `mbMotion_`, `mbIntra4x4Modes_`, `qp_`,
CBP, etc.). When an MB context is read by deblock, the CPU must fetch
cache lines from multiple arrays → more L1 misses.

## Problem

[decoder.hpp](../../../components/sub0h264/src/decoder.hpp) members:
- `std::vector<uint8_t> nnzLuma_[totalMbs * 16]`
- `std::vector<uint8_t> nnzCb_[totalMbs * 4]`
- `std::vector<uint8_t> nnzCr_[totalMbs * 4]`
- `std::vector<MbMotionInfo> mbMotion_[totalMbs * 16]`
- `std::vector<uint8_t> mbIntra4x4Modes_[totalMbs * 16]`
- various flags arrays

Each MB's context is scattered across ~6 arrays. Deblocking a single MB
pulls from all of them.

## Options

### Option A — `MbContext` struct, one array of structs
Pack every per-MB field into a single struct, one struct per MB:
```cpp
struct MbContext {
    uint8_t nnzLuma[16];
    uint8_t nnzCb[4];
    uint8_t nnzCr[4];
    MbMotionInfo motion[16];    // or packed
    uint8_t intra4x4Modes[16];
    int8_t qp;
    uint8_t cbp;
    uint8_t flags;               // intra, skip, transform8x8, …
    uint8_t bs[16];              // from L2.3
};
// std::array<MbContext, cMaxMbs> context_;
```

- **Pro:** single cache-line burst to fetch the MB's full context
- **Pro:** composes cleanly with [L1.3 static arrays](L1.3_static_context_arrays.md)
- **Con:** large struct (~100+ bytes); may waste cache on reads that
  only need a few fields (e.g. entropy decode that only reads NNZ)
- **Con:** more invasive refactor — many call-sites to update

### Option B — Parallel arrays but co-located (`struct { Array<...> a, b, c; }`)
Keep separate arrays but guarantee they're allocated adjacently.

- **Pro:** minimal call-site change
- **Con:** doesn't help cache behaviour since each array is still a
  separate stride

### Option C — Hybrid: hot fields in one struct, cold fields separate
Group only the deblock-critical fields (nnz, qp, flags, bs) into a small
hot struct. Leave motion/modes separate.

- **Pro:** smaller hot struct, better cache density for the deblock pass
- **Con:** more complex — two array types to maintain

## Chosen approach: **Option C (hybrid)** once [L1.3](L1.3_static_context_arrays.md) lands

Reasoning: Option A's big struct spreads the deblock-critical data across
more cache lines than strictly necessary. Hot struct (~32 B) fits in a
single cache line. Motion info is larger and accessed by MC / MV
prediction which have their own access patterns.

```cpp
struct MbHotContext {
    uint8_t nnzLuma[16];   // 16 B
    uint8_t nnzCb[4];      // 4 B
    uint8_t nnzCr[4];      // 4 B
    int8_t qp;             // 1 B
    uint8_t cbp;           // 1 B
    uint8_t flags;         // 1 B (intra/skip/...)
    uint8_t bs[16];        // 16 B
    // total 43 B — pad to 64 B cache line
    uint8_t pad[21];
};
```

Keep `mbMotion_` and `mbIntra4x4Modes_` as separate arrays (motion is
accessed by MC, different hot loop).

## Implementation

**Files:** [components/sub0h264/src/decoder.hpp](../../../components/sub0h264/src/decoder.hpp)

**Changes:**
1. Define `MbHotContext` struct (aligned to 64 B).
2. Replace `nnzLuma_`, `nnzCb_`, `nnzCr_`, `qp_`, `cbp_`, flags arrays
   with `std::array<MbHotContext, cMaxMbs> mbHotCtx_`.
3. Add convenience accessors: `nnzLuma(mb,i)` etc. that resolve through
   the struct.
4. Update deblocking and CABAC neighbour code to read from the struct.
5. Keep `mbMotion_` and `mbIntra4x4Modes_` as separate arrays.

## Validation

- 99 dB bit-exact on golden fixtures
- All `ctest --preset default` passes
- Bench: deblock-bound fixtures +0.3 fps; overall compound benefit

## Risks & mitigations

- **Struct layout creep:** every added field expands the struct, possibly
  crossing the 64 B boundary. Add `static_assert(sizeof(MbHotContext) ==
  64)` to catch accidental growth.
- **Refactor breadth:** ~30 call-sites to update. Do this in a branch;
  measure PSNR at each commit.

## Follow-ups / compounding opportunities

- Best done AFTER [L1.3 static arrays](L1.3_static_context_arrays.md) —
  the new struct array replaces the vectors directly.
- Enables [L2.3 BS cache](L2.3_deblock_bs_cache.md) as a field of the
  hot struct (zero additional allocation cost).
- Sets up easier PIE paths that load multiple context fields in one
  vector load.
