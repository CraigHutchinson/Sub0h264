# L4.1 — Branchless `clipU8`

**Layer:** L4 Implementation | **Expected gain:** +2–3% overall | **Risk:** Low | **Complexity:** S | **Status:** pending

## Context

`clipU8` is called thousands of times per frame — every pixel output from
IDCT, MC, intra prediction, deblocking passes through it. The current
implementation is a nested ternary that compiles to two conditional
branches. On the ESP32-P4 in-order core, branch mispredictions cost 2–6
cycles each; with random content (ball benchmark), branches are often
unpredictable.

## Problem

[transform.hpp](../../../components/sub0h264/src/transform.hpp) line 45:
```cpp
inline constexpr int32_t clipU8(int32_t val) noexcept
{
    return val < 0 ? 0 : (val > 255 ? 255 : val);
}
```

Modern x86 compilers often lower this to `CMOVcc` (branchless) but RISC-V
in-order cores without a proper saturating unsigned-pack instruction can
leave this as two compare-and-branches.

## Options

### Option A — Bit-manipulation clip
```cpp
inline constexpr int32_t clipU8(int32_t val) noexcept
{
    // If val > 255, set to 255; if val < 0, set to 0 — via signed arithmetic.
    val &= ~(val >> 31);          // clear to 0 if negative (sign-extend mask)
    return val | ((255 - val) >> 31);  // saturate to 255 if > 255
}
```
(Actual correct branchless formulation varies — needs verification.)

- **Pro:** branchless; constant-time; ~3 instructions on RISC-V
- **Con:** relies on signed >> behaviour (implementation-defined in
  pre-C++20; well-defined in C++20 as arithmetic shift)
- **Con:** subtle; needs a unit test covering boundary inputs

### Option B — `std::clamp(val, 0, 255)`
Standard library.

- **Pro:** idiomatic
- **Con:** in practice compiles to the same nested branches on ESP32
  toolchain; not a win unless compiler improves

### Option C — GCC builtin `__builtin_clamp`
Not universally available.

- **Con:** tooling dependency

### Option D — Use `(uint32_t)val` bit trick
```cpp
return (uint32_t)val > 255 ? ((int32_t)val < 0 ? 0 : 255) : val;
```
The unsigned cast folds the < 0 and > 255 cases into one compare.

- **Pro:** reads more clearly
- **Con:** still a conditional; compiler-dependent

## Chosen approach: **Option A with explicit test**

The bit-manipulation form is proven in h264/h265 reference codecs and
compilers lower it consistently to branchless code on RISC-V.

```cpp
inline constexpr int32_t clipU8(int32_t val) noexcept
{
    // Saturate to [0, 255] without branches.
    // Step 1: if val < 0, set to 0 (mask out negative).
    val = val & ~(val >> 31);
    // Step 2: if val > 255, set to 255.
    val |= (255 - val) >> 31;
    return val & 0xFF;
}
```

## Implementation

**Files:** [components/sub0h264/src/transform.hpp](../../../components/sub0h264/src/transform.hpp) line 45

**Changes:**
1. Replace current ternary-based `clipU8` with branchless form.
2. Mark `[[gnu::always_inline]]` (it's already `inline constexpr` — should

## Validation

- 99 dB bit-exact on ALL golden fixtures (any bug here shows up instantly)
- Compile-time `static_assert`: `clipU8(-1) == 0`, `clipU8(0) == 0`,
  `clipU8(128) == 128`, `clipU8(255) == 255`, `clipU8(256) == 255`,
  `clipU8(INT32_MAX) == 255`, `clipU8(INT32_MIN) == 0`.
- Exhaustive unit test over `[-1024, 1280]`.

## Risks & mitigations

- **Signed-shift UB:** `val >> 31` on a negative signed integer is
  implementation-defined before C++20. C++20 defines it as arithmetic
  (sign-extending). Our codebase uses C++17 — explicitly test on GCC
  and MSVC; they both do arithmetic shift in practice.
  Alternative: use `(val & 0x80000000) ? -1 : 0` for the mask.

- **IDCT output range:** 9-bit intermediate (`[-256, 511]`) fits in
  int32 easily; the branchless clip handles both bounds.
