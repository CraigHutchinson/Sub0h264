# L4.2 — Deblocking inline row pointers

**Layer:** L4 Implementation | **Expected gain:** +5–10% of deblock time | **Risk:** Low | **Complexity:** S | **Status:** pending

## Context

The deblock filter reads pixels via `frame.yRow(r) + x` on every filter
tap. `yRow(r)` is a multiply-add (`base + r * stride`). For a single
vertical edge, the filter runs on 4 rows × 4 or 8 taps = 16–32 `yRow`
calls; each recomputes the same stride × row multiplication.

## Problem

[deblock.hpp](../../../components/sub0h264/src/deblock.hpp) lines 369 and
500–507 — filter loops call `yRow(pixY + r)` per row. The compiler may
CSE this across taps in one invocation but won't across consecutive edge
invocations (rows are the same, only `x` changes).

Example current pattern:
```cpp
for (int r = 0; r < 4; ++r) {
    uint8_t* row = frame.yRow(pixY + r) + edgeX;  // mul + add
    filter(row);
}
```

## Options

### Option A — Precompute row pointer table once per edge column
```cpp
uint8_t* rows[16];
for (int r = 0; r < 16; ++r)
    rows[r] = frame.yRow(pixY + r) + edgeX;
// use rows[r] in inner loop
```

- **Pro:** 16 multiplies replaced with 16 adds (stride is constant)
- **Pro:** compiler can keep `rows[]` in registers for hot loops
- **Con:** 128 B of stack for the pointer table; trivial

### Option B — Do the row pointer math inline, rely on compiler CSE
Hope for the best with `[[gnu::always_inline]]` and aggressive `-O2`.

- **Pro:** no code change
- **Con:** compiler may not CSE across function boundaries; measured
  improvement (profile) shows the multiplies are still there on some
  compilers

### Option C — Refactor deblock to take a pre-computed `RowTable` struct
Encapsulate the table in a helper that the MB-level deblock fills once.

- **Pro:** cleaner API, reusable across luma/chroma
- **Con:** slightly more structural change

## Chosen approach: **Option A inline, or Option C if it reads cleaner**

Start with Option A per-function. If the resulting duplication is ugly,
promote to Option C (row table helper).

## Implementation

**Files:** [components/sub0h264/src/deblock.hpp](../../../components/sub0h264/src/deblock.hpp)

**Changes:**
1. At the entry of `deblockMbLuma`:
   ```cpp
   alignas(16) uint8_t* yRows[16];
   for (int r = 0; r < 16; ++r)
       yRows[r] = frame.yRow(mbPixY + r);
   ```
2. Pass `yRows` (not `frame`) to all internal edge filters.
3. Filter functions take `uint8_t** rows, int rowStart, int edgeX, …`.
4. Same for chroma with 8-row table.

## Validation

- 99 dB bit-exact on golden fixtures
- Bench: deblock-bound fixtures (Scroll, Tapo) + ≥ 0.2 fps
- Compile and assembly-inspect one filter function to confirm the
  `rows[r]` stays in a register

## Risks & mitigations

- **Row table correctness at frame edge:** trivial; `yRow(r)` bounds-
  checks are the caller's responsibility, no change in semantics
- **Interaction with [L2.4 transpose](L2.4_horizontal_edge_transpose.md):**
  transpose reads from row pointers too; use the row table as input to
  the transpose helper

## Follow-ups / compounding opportunities

- Pairs with [L2.4 horizontal transpose](L2.4_horizontal_edge_transpose.md)
  — the row table feeds into the transpose step
- Sets up PIE vector loads ([L5.3 catalog](L5.3_pie_simd_catalog.md))
  where a row pointer is the natural vector base
