# L4.3 — CAVLC `coeff_token` LUT

**Layer:** L4 Implementation | **Expected gain:** +5–8% of entropy time | **Risk:** Low | **Complexity:** M | **Status:** pending

## Context

CAVLC entropy decoding begins with `coeff_token` which encodes both
`trailingOnes` and `totalCoeff`. The spec table (§9.2.3.1 Tables 9-5..9-10)
has up to 62 entries per `nC` value (neighbour-derived context index).
Current implementation does a linear scan of entries comparing against
the next few bits in the stream.

## Problem

[cavlc.hpp](../../../components/sub0h264/src/cavlc.hpp) `matchCoeffTokenTable`
(lines ~72–110) iterates entries linearly. For every coeff_token decode
(once per 4x4 block of a coded MB), the worst case is ~62 comparisons.
Average maybe ~10 with a well-pruned table; still a sequential dependency
chain on the in-order CPU.

## Options

### Option A — Binary search by code length
Group entries by length (1–16 bits), then binary search within the length
bucket. ~6 comparisons max.

- **Pro:** bounded worst case
- **Pro:** minimal memory (same table, one index array per nC)
- **Con:** still branchy; CPU-wise on par with a well-optimised linear
  scan for short codes (common case is 2–4 bits)

### Option B — Two-level LUT: first 6 bits → length + partial result
Read 6 bits, look up in a 64-entry table. If result fits in 6 bits we're
done; else fall back to full match for the rare long codes.

- **Pro:** zero branches for ~95% of cases
- **Con:** ~256 B table per nC × 4 nC buckets = ~1 KB; not free but fits
  in L1

### Option C — Full 8-bit LUT with in-table "need-more-bits" marker
Peek 8 bits, table entry says either "decoded (length, result)" or
"consume some bits, look up next table".

- **Pro:** fastest
- **Con:** table design complexity; more memory

## Chosen approach: **Option B — 6-bit peek table**

The H.264 coeff_token codes are 1–16 bits; most are ≤ 6 bits. A 6-bit
peek table covers the common case in one read, falling back to binary
search for the long tail.

```cpp
struct CoeffTokenEntry {
    uint8_t length;   // code length in bits
    uint8_t result;   // packed (trailingOnes, totalCoeff)
};
// Per nC bucket: lookup[0..63] = 6-bit peek result, or
// {length=0xFF, result=index-into-fallback-table}.
```

## Implementation

**Files:** [components/sub0h264/src/cavlc.hpp](../../../components/sub0h264/src/cavlc.hpp)

**Changes:**
1. Generate per-nC 64-entry peek tables at build time via `constexpr`
   derivation from the spec tables.
2. `matchCoeffTokenTable` becomes:
   - Peek 6 bits from bitstream
   - Look up in peek table
   - If length > 6, fall through to the existing linear scan or a
     smaller fallback table
   - Else consume `length` bits, return `result`
3. Add a `static_assert` / compile-time test that the peek table is
   derived consistently from the original table.

## Validation

- 99 dB bit-exact on CAVLC fixtures (most of the benchmark suite)
- Existing CAVLC unit tests in `tests/test_cavlc.cpp` pass unchanged
- Bench: Baseline (CAVLC-only) fixtures +3–5% fps

## Risks & mitigations

- **Table derivation correctness:** easy to get wrong. Write a
  `constexpr` generator that takes the spec table as input and produces
  the peek table; cross-check at compile time.
- **Endianness / bit order:** CAVLC reads MSB-first. Peek 6 bits returns
  the 6 MSBs of the next byte(s). Careful with the peek API.
- **Long codes (>6 bits) are rare:** in the fallback path, the linear
  scan already works — keep it as-is, just skip entries of length ≤ 6.

## Follow-ups / compounding opportunities

- Same pattern applies to other CAVLC tables (level prefix, total_zeros,
  run_before) — each ~30% of entropy time. Ticket separately if
  useful.
- Paired compile-time table generation aligns with the "constexpr table
  from spec formula" rule in [CLAUDE.md](../../../CLAUDE.md#constexpr-table-generation-from-spec-formulas)
