# L4.4 — Fully unrolled 4×4 IDCT

**Layer:** L4 Implementation | **Expected gain:** +3–5% of transform time | **Risk:** Low | **Complexity:** S | **Status:** pending

## Context

The 4×4 inverse IDCT runs a 4-iteration loop twice (column pass then row
pass) per 4×4 block. Each MB has up to 16 × 4×4 blocks of luma +
8 × 4×4 blocks of chroma — ~24 IDCT invocations per MB. The loops are
short and regular; the compiler usually unrolls them, but on ESP32-P4's
in-order core with limited register pressure, explicit unrolling can
eliminate any remaining loop overhead and help scheduler visibility.

## Problem

[transform.hpp](../../../components/sub0h264/src/transform.hpp)
`inverseDct4x4AddPred` has two `for (uint32_t i = 0U; i < 4U; ++i)`
loops — column pass then row pass. Compilers at `-O2` unroll these but
inconsistently across toolchains.

## Options

### Option A — Hand-unrolled butterfly with explicit variables
Replace loops with straight-line code:
```cpp
int32_t r0c0 = block[0] + block[8];  // e0 = d0 + d2
int32_t r1c0 = block[0] - block[8];  // e1 = d0 - d2
// ... 8 butterfly operations ...
out[0][0] = clipU8(pred[0] + ((r0c0 + r0c3) >> 6));
// ... 16 output stores ...
```

- **Pro:** explicit — no dependence on compiler unrolling
- **Pro:** easier to schedule by hand
- **Con:** ~80 lines of straight-line code; less readable but stable

### Option B — `#pragma GCC unroll 4` on the loops
One-line compiler hint.

- **Pro:** minimum change
- **Con:** pragma availability varies; MSVC ignores GCC pragmas; not
  portable

### Option C — `[[gnu::always_inline]]` + `[[gnu::hot]]` + `__builtin_unrolled_loop()`
- **Pro:** stronger compiler nudge
- **Con:** doesn't guarantee unrolling; still compiler-dependent

## Chosen approach: **Option A (explicit unroll)**

For a function this hot, explicit beats compiler luck. It also makes the
function a natural candidate for PIE conversion later — the 8-wide SIMD
layout is clearer in straight-line code.

## Implementation

**Files:** [components/sub0h264/src/transform.hpp](../../../components/sub0h264/src/transform.hpp)

**Changes:**
1. Replace `inverseDct4x4AddPred` body with fully unrolled butterfly.
2. Add `[[gnu::always_inline]] [[gnu::hot]]` attributes.
3. Keep the existing implementation in a `#if 0` / fallback for
   bit-exactness diffing during development.
4. Optionally add a `-DSUB0H264_IDCT_UNROLL=0` macro to fall back to
   looped implementation for debug builds.

## Validation

- 99 dB bit-exact on ALL golden fixtures (every MB with residual runs
  IDCT — this is the most-exercised function in the codebase)
- Existing IDCT unit tests in `tests/test_transform.cpp` pass
- Bench: all fixtures +0.5–1% fps; Still and Ball more noticeable

## Risks & mitigations

- **Reordering subtly changes result (e.g. `+` vs `-` order with signed
  overflow):** spec arithmetic uses 32-bit signed so overflow isn't
  defined; any reordering must match the spec expression order exactly.
  Keep the original reference in a test, compare outputs on random
  input blocks.
- **Code size:** ~80 lines per direction × 2 directions × 2 variants
  (add vs reconstruct) could balloon. Measure `.text` before/after.

## Follow-ups / compounding opportunities

- Unrolled straight-line code maps cleanly to PIE 8-wide int16
  instructions (see [L5.3 catalog](L5.3_pie_simd_catalog.md))
- Combine with [L4.1 branchless clipU8](L4.1_branchless_clipu8.md) —
  the 16 final clipU8 calls can be unrolled + branchless together
- Same pattern applicable to 8×8 IDCT; evaluate separately
