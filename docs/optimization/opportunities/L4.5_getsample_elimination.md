# L4.5 — `getSample` elimination via L3.1 prefetch

**Layer:** L4 Implementation | **Expected gain:** +1.5 fps (compounds with L3.1) | **Risk:** Low | **Complexity:** S | **Status:** pending

## Context

Every 6-tap luma MC filter call invokes `getSample(ref, x, y)`, which
clamps `x,y` to `[0, w-1] × [0, h-1]` before the read. For a 16×16 block
at a diagonal position: 256 output pixels × 36 taps = 9 216 `getSample`
calls, each with 4 branches (low-X, high-X, low-Y, high-Y).

## Problem

[inter_pred.hpp](../../../components/sub0h264/src/inter_pred.hpp) —
`getSample` is defensive against frame-edge MVs. Once [L3.1 reference
prefetch](L3.1_reference_prefetch.md) copies the reference region with
margin into a stack scratch buffer, all coordinates in the filter inner
loop are guaranteed in-bounds (clamping was done during the prefetch
copy). The defensive `getSample` is now dead code.

## Options

### Option A — Add a separate filter variant that reads from prefetch scratch
The existing `getSample`-based filter stays as a fallback; a new
`filter6TapFromScratch(const uint8_t* scratch, int stride, …)` takes
the scratch pointer and uses direct indexing.

- **Pro:** clean separation; no risk to the fallback path
- **Con:** two filter implementations to maintain (transitional only)

### Option B — Rewrite the filter to always take a pre-fetched region
After L3.1 lands, every MC path passes scratch to the filter. `getSample`
is deleted.

- **Pro:** single implementation
- **Con:** must be done after L3.1; sequencing matters

## Chosen approach: **Option B, sequenced after L3.1**

Once [L3.1](L3.1_reference_prefetch.md) is the default, `getSample` is
unreachable. Delete it.

## Implementation

**Files:** [components/sub0h264/src/inter_pred.hpp](../../../components/sub0h264/src/inter_pred.hpp)

**Changes:**
1. Prerequisite: [L3.1](L3.1_reference_prefetch.md) merged.
2. Rewrite `filter6Tap(...)` / `filter8Tap(...)` to take
   `const uint8_t* scratch, int stride` and use direct indexing
   `scratch[row * stride + col]`. Remove all `getSample` calls.
3. Delete `getSample` entirely.
4. Check margin in scratch: row 0..20 is usable (for 16-high block with
   5-pixel top + bottom margin); ensure no filter reads outside this
   range.

## Validation

- 99 dB bit-exact on golden fixtures
- Frame-edge unit tests (MB at (0,0) with MV (-3, -3), etc.) — same
  validation as [L3.1](L3.1_reference_prefetch.md)
- Bench: Ball-Base-640 fps + ≥ 1.5 **on top of** L3.1 improvement

## Risks & mitigations

- **Silent out-of-bounds read if margin is miscomputed:** add
  debug-only bounds assertions in the filter (strip in Release but
  verify in unit tests). An assert at the top of each filter like:
  `assert(row + tap < margin_bound)` catches this.
- **Chroma path:** chroma uses bilinear (2-tap) with smaller margin.
  Separate scratch / filter pair.

## Follow-ups / compounding opportunities

- Direct dependency: [L3.1 reference prefetch](L3.1_reference_prefetch.md)
- Enables [L2.1 diagonal MC 2-pass](L2.1_diagonal_mc_2pass.md) — a clean
  scratch-based filter is the natural input to the 2-pass separable form
- Prereq for PIE vectorisation ([L5.3 catalog](L5.3_pie_simd_catalog.md))
