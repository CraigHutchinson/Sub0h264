# L5.1 — Compiler hints + alignment

**Layer:** L5 Micro | **Expected gain:** +1–2% | **Risk:** Low | **Complexity:** S | **Status:** pending

## Context

The ESP32-P4 toolchain (Xtensa LX7 + RISC-V glue) is mature but ESP-IDF
defaults don't aggressively inline / alias-analyse hot paths. A small set
of compiler hints can nudge the optimiser at the hottest functions
without touching actual logic.

## Problem

No organised hint set today. Specifically:
- MC / deblock pointer parameters don't use `__restrict__`; the compiler
  assumes potential aliasing and emits redundant loads/stores.
- Hot functions (IDCT, MC filter, deblock filter) don't have
  `[[gnu::hot]]` or `[[gnu::always_inline]]` attributes.
- Stack scratch buffers for MC ([L3.1](L3.1_reference_prefetch.md))
  don't use `alignas(32)` for cache-line alignment on ESP32-P4.
- CABAC MPS path (decoded ~90% of the time) doesn't use
  `__builtin_expect` to help the branch predictor.

## Options

### Option A — Apply hints broadly to documented hot paths
Identify the top-N functions by profile; annotate each with the full set:
`[[gnu::hot]] [[gnu::always_inline]]`, `__restrict__` on pointer params,
`alignas(32)` on key scratch buffers, `__builtin_expect(mps, 1)` in CABAC.

- **Pro:** documented, local, reversible
- **Pro:** `[[gnu::hot]]` also gives the linker locality hint
- **Con:** little effect if the compiler already did the right thing

### Option B — `#pragma GCC optimize` at function level
Force specific `-O3` or loop-vectorise on hot functions while keeping the
rest at the project default.

- **Pro:** can unlock benefits without global compiler flag changes
- **Con:** pragma syntax varies; not portable; easy to accidentally
  enable UB-prone optimisations (strict-aliasing)

### Option C — Profile-guided optimisation (PGO)
- **Pro:** measured improvement
- **Con:** ESP-IDF build system doesn't trivially support PGO; complex
  toolchain work

## Chosen approach: **Option A**

Annotation-based hints are the lowest-risk, highest-portability wins.
Implement in one pass across the identified hot functions.

## Implementation

**Files** (hot functions per profile):
- [inter_pred.hpp](../../../components/sub0h264/src/inter_pred.hpp) — filter functions, `__restrict__` on src/dst pointers, `[[gnu::hot]]`, `alignas(32)` on stack scratch
- [transform.hpp](../../../components/sub0h264/src/transform.hpp) — IDCT, `clipU8`
- [deblock.hpp](../../../components/sub0h264/src/deblock.hpp) — edge filter functions
- [cabac.hpp](../../../components/sub0h264/src/cabac.hpp) — `decodeBin` MPS path `__builtin_expect(mps, 1)`
- [cavlc.hpp](../../../components/sub0h264/src/cavlc.hpp) — hot decode functions

**Convention to apply:**
```cpp
[[gnu::hot]] [[gnu::always_inline]]
inline void filter6Tap(const uint8_t* __restrict__ src, int stride,
                       uint8_t* __restrict__ dst, ...) noexcept { ... }
```

And for CABAC:
```cpp
if (__builtin_expect(bit == context.mps, 1)) {
    // fast path
}
```

And for stack scratch in MC:
```cpp
alignas(32) uint8_t scratch[21 * 21];
```

## Validation

- 99 dB bit-exact on all fixtures (attributes don't affect semantics)
- `ctest --preset default` passes
- Bench: small improvement across all fixtures; compound benefit when
  combined with the algorithmic opts

## Risks & mitigations

- **`__restrict__` assumes no aliasing:** if the caller passes the same
  pointer for src and dst (unusual but possible), behaviour is UB. Audit
  call sites. Our MC/filter/IDCT functions never share src and dst in
  practice; assert at call-site if needed.
- **MSVC compatibility:** `[[gnu::...]]` attributes degrade gracefully
  (ignored). `__restrict__` → use `__restrict` (no trailing underscore).
  Define a portable `SUB0H264_RESTRICT` macro.
- **Attribute spam:** keep to ~10 truly hot functions. Over-annotation
  loses its signal.

## Follow-ups / compounding opportunities

- Compounds with every other optimisation by giving the compiler more
  signal
- Proper PGO (Option C) is a future win on ESP-IDF once tooling supports it
