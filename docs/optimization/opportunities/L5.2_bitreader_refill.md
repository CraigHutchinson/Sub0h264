# L5.2 — BitReader cache refill

**Layer:** L5 Micro | **Expected gain:** +1–2% of entropy time | **Risk:** Low | **Complexity:** S | **Status:** pending

## Context

`BitReader` holds a 64-bit cache of pending bits. When the cache runs
low, it refills from the RBSP buffer. The current refill reads byte-by-byte
in a loop — on ESP32-P4 RISC-V, this is 8 sequential loads + 8 shifts to
assemble a word.

Modern compilers lower `memcpy(8 bytes)` to inline word loads
(`lw` + `lw` on RV32, or `ld` on RV64), and `__builtin_bswap64` to a
single byte-swap instruction (or optimized inline if no native swap).
The byte-by-byte loop defeats both optimisations.

## Problem

[bitstream.hpp](../../../components/sub0h264/src/bitstream.hpp)
`refillCache` (around line 252):
```cpp
while (bitsLeft_ <= 56 && bytePos_ < endPos_) {
    cache_ |= static_cast<uint64_t>(data_[bytePos_++]) << (56 - bitsLeft_);
    bitsLeft_ += 8;
}
```

8 iterations × (load, shift, or, increment, compare) vs what should be
one `memcpy + bswap + shift`.

## Options

### Option A — `memcpy(&chunk, data + pos, 8); cache_ |= bswap64(chunk) >> bitsLeft_`
Replace the loop with one 8-byte read and a bswap.

- **Pro:** compiles to 2–3 instructions on RISC-V with the B extension
  (no B extension → inline byte-swap, still fewer ops than the loop)
- **Con:** need bounds-check: if fewer than 8 bytes remain, fall back
  to the byte loop

### Option B — Unrolled 4× byte loop with manual shift
Middle ground: 4 iterations of unrolled load+or+shift.

- **Pro:** no `memcpy` dependency
- **Con:** still worse than a single word load

### Option C — Expose `refillCache` as `[[gnu::hot]] [[gnu::always_inline]]`
No algorithm change.

- **Pro:** zero risk
- **Con:** Minor gain; the byte loop is the bottleneck

## Chosen approach: **Option A (memcpy + bswap fast path)** with byte-loop fallback

```cpp
if (__builtin_expect(endPos_ - bytePos_ >= 8, 1)) {
    uint64_t chunk;
    std::memcpy(&chunk, data_ + bytePos_, 8);
    chunk = __builtin_bswap64(chunk);          // we want MSB-first
    // shift chunk to align with current cache
    cache_ |= chunk >> bitsLeft_;              // or similar
    bytePos_ += (64 - bitsLeft_) / 8;
    bitsLeft_ |= 56;  // round up to next 8
} else {
    // fallback byte loop
    while (bitsLeft_ <= 56 && bytePos_ < endPos_) { ... }
}
```

## Implementation

**Files:** [components/sub0h264/src/bitstream.hpp](../../../components/sub0h264/src/bitstream.hpp)

**Changes:**
1. Add fast path in `refillCache` using `memcpy(8) + bswap64`.
2. Keep the byte loop as fallback for end-of-buffer case.
3. Mark `refillCache` `[[gnu::hot]] [[gnu::always_inline]]`.
4. Verify `__builtin_bswap64` availability (GCC/Clang/MSVC all have it;
   `_byteswap_uint64` on MSVC).

## Validation

- 99 dB bit-exact on golden fixtures (CABAC and CAVLC both exercise
  BitReader extensively)
- Existing bitstream unit tests pass
- Unit test: end-of-buffer refill (last 1–7 bytes) must hit the fallback
  path and produce the same bit sequence
- Bench: CABAC-heavy fixtures (Tapo, High) +0.5% fps

## Risks & mitigations

- **Endianness:** `memcpy` reads in memory order; `bswap64` converts to
  big-endian (MSB-first) semantics that CABAC/CAVLC expect. Verify
  behaviour on a known bit pattern at unit test level.
- **Alignment:** `memcpy` of 8 bytes with an unaligned source is
  well-defined on x86 but can be slower on older RISC-V without the
  `Zicbom` or `Zabha` extensions. ESP32-P4 handles unaligned loads
  efficiently; measure if concerned.
- **End-of-buffer boundary:** most critical — make sure the `>= 8`
  check is correct and the fallback handles 0–7 remaining bytes.

## Follow-ups / compounding opportunities

- Same pattern applies to other bitstream helpers (`readBits`,
