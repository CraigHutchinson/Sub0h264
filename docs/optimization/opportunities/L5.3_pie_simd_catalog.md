# L5.3 — PIE / SIMD opportunity catalogue (DOCUMENT ONLY)

**Layer:** L5 Micro | **Expected gain:** 4–8× per hot kernel when enabled | **Risk:** N/A (no impl) | **Complexity:** L | **Status:** catalogue, not implemented

## Context

ESP32-P4 includes the PIE (Processor Instruction Extensions) which
provide 128-bit vector operations: 16× uint8, 8× int16, 4× int32. For
H.264 kernels with data-parallel structure — MC filters, IDCT, deblock,
clip — PIE speedups of 4–8× per kernel are realistic. PIE adoption isn't
in scope for this plan (tooling / workflow not ready), but the catalogue
documents which kernels should be converted first, in what order, and
what data-layout prerequisites each one needs.

## Purpose of this document

- **Do not implement PIE kernels as part of the optimisation plan.**
- Capture, for each hot kernel, the expected PIE op-set and speedup.
- Name the prerequisites (data layout, alignment, scratch buffers) each
  kernel needs before PIE lowering becomes mechanical.
- Prioritise kernels for a future PIE effort based on fraction-of-time.

## Priority for a future PIE phase

1. **`lumaMotionComp` horizontal** — single biggest win (30% of decode
   × 4–8×). Implement this kernel first. All other PIE work delivers less
   benefit.
2. **`lumaMotionComp` vertical** — follow-up; depends on the same
   data-layout foundation.
3. **`chromaMotionComp`** — smaller share of time but structurally
   simpler; good second-priority test of the toolchain.
4. **Deblock filter (luma/chroma strong + weak)** — deblock is 13.8% of
   decode; per-kernel speedup 2–4× means meaningful gain.
5. **`inverseDct4x4AddPred`, `inverseDct8x8AddPred`** — medium priority;
   the existing C code is already close to the scheduling limit of
   scalar cores. Vectorisation unlocks the saturated-pack output step.
6. **Full-pel memcpy** — trivial; falls out of [L3.2 alignment](L3.2_frame_alignment.md).

## Implementation notes for a future PIE effort

- PIE intrinsics are defined in [ESP-IDF's pie.h](https://docs.espressif.com/projects/esp-idf/).
  A thin wrapper layer (`components/sub0h264/src/simd.hpp`) should
  expose platform-agnostic helpers that compile to PIE on ESP32-P4 and
  to SSE2 on x86 for the desktop shootout.
- Each PIE kernel MUST have a scalar fallback selected at compile time
  by a macro (e.g. `SUB0H264_USE_PIE`). Unit tests run both paths and
  assert bit-exact equivalence.
- Use `alignas(16)` on intermediate buffers and load source data into
  16-byte-aligned stack scratch. [L3.2 alignment](L3.2_frame_alignment.md)
  is a hard prerequisite.

## Validation framework (when PIE is enabled)

- **Required:** bit-exact equivalence between scalar and PIE paths for
  every kernel, enforced at unit-test level (random input vectors).
- **Required:** per-function bench comparing scalar vs PIE cycles.
- **Required:** Ball-Base-640 fps must hit ≥ 25 with PIE enabled (the
  headline target).

## When to start PIE work

After Phases 3–13 of the optimisation plan land, remeasure ESP32 fps.
If Ball-Base-640 < 25 fps, PIE becomes the next phase. If ≥ 25 fps, PIE
is not needed for the MVP — defer to a later performance band target.
